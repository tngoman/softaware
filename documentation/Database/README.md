# Database Module — Overview

**Version:** 1.0.0  
**Last Updated:** 2026-03-05

---

## 1. Module Overview

### Purpose

The Database module is the foundational persistence layer of the SoftAware platform. It manages **two independent storage engines** that serve complementary purposes:

1. **MySQL 8.x** — The primary relational store for all structured application data (users, contacts, invoices, subscriptions, assistants, teams, updates, cases, etc.). All business logic routes through MySQL via a custom thin-wrapper `db` helper built on `mysql2/promise`.

2. **sqlite-vec** — A specialised embedded vector database for semantic similarity search, powering the Assistants module's RAG (Retrieval-Augmented Generation) pipeline. Uses [sqlite-vec](https://github.com/asg017/sqlite-vec) loaded into `better-sqlite3`, storing 768-dimensional float32 embeddings generated by the `nomic-embed-text` model.

### Business Value

- **Single connection pool** for all MySQL access — no ORM overhead, raw SQL with parameterised queries
- **Zero-dependency vector search** — sqlite-vec runs in-process (no external vector DB to manage)
- **Dual-storage ingestion** — knowledge chunks written to both MySQL (relational queries) and sqlite-vec (KNN search)
- **Transaction support** via pool connections with automatic rollback on failure
- **Type-safe interfaces** — TypeScript interfaces for every table domain in dedicated type files
- **Idempotent migrations** — each migration checks before altering, preventing re-run failures

### Key Statistics

| Metric | Value |
|--------|-------|
| MySQL tables | 78 (active + legacy `tb_*` PHP imports) |
| Active tables | 63 (excluding `tb_*` prefix) |
| Legacy tables | 15 (`tb_*` — PHP-era data, read-only) |
| sqlite-vec tables | 2 (`knowledge_chunks`, `knowledge_vectors`) |
| Type definition files | 2 (`businessTypes.ts`, `updatesTypes.ts`) |
| Migration files | 11 (10 TypeScript + 1 raw SQL) |
| Connection pool limit | 10 connections |
| Collation (primary) | `utf8mb4_unicode_ci` / `utf8mb4_0900_ai_ci` (mixed) |
| Vector dimensions | 768 (nomic-embed-text, float32) |
| sqlite-vec DB size | ~4 KB (initial), grows with ingested knowledge |

## 4. Two-Engine Design — Why MySQL + sqlite-vec?

| Concern | MySQL | sqlite-vec |
|---------|-------|------------|
| Relational queries (JOINs, WHERE, ORDER BY) | ✅ Primary use | ❌ Not suited |
| ACID transactions | ✅ InnoDB + pool | ✅ WAL + transactions |
| Full-text search | 🟡 Possible but slow | ❌ Not available |
| K-nearest-neighbor vector search | ❌ No native support | ✅ vec0 MATCH + k |
| Embedding storage | 🟡 JSON column (slow scan) | ✅ Float32 binary (fast) |
| Operational overhead | Server process, backups, credentials | Zero — single file, in-process |
| Concurrent writes | ✅ Connection pool (10) | 🟡 WAL (one writer at a time) |
| Use case | All structured data | RAG retrieval only |

The dual-storage design means **every ingested knowledge chunk is written to both stores**: MySQL (`assistant_knowledge`) for relational access and recategorization, and sqlite-vec (`knowledge_chunks` + `knowledge_vectors`) for the fast KNN search that powers the assistant chat's RAG pipeline.
